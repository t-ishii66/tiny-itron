# start.s

対象ファイル: `i386/start.s`

## 概要

第二段ブートローダー。16 ビットリアルモードから 32 ビットプロテクトモードへの移行を行う。boot.s の `ljmp $0x300, $0` により物理アドレス 0x3000 に配置されたこのコードに制御が渡される。A20 ライン有効化、GDTR/IDTR のロード、CR0 の PE ビット設定によるプロテクトモード遷移、32 ビットコードセグメントへの far jump を行う。

AP (Application Processor、CPU 1) の起動時にも SIPI (Startup IPI) のベクタが 0x03 (= 物理 0x3000) に設定されるため、このコードが再実行される。AP は同じ手順でプロテクトモードに移行した後、run.s に到達して `cpu_num` の値で BSP/AP を区別する。

## 定数・マクロ

| 名前 | 値 | 説明 |
|------|------|------|
| `b_base` | `0x07C0` | ブートセクタのセグメントアドレス (未使用) |
| `s_base` | `0x0300` | start.s 自身のセグメントアドレス (0x0300:0000 = 物理 0x3000) |
| `k_base` | `0x3400` | カーネルエントリポイント (リニアアドレス。プロテクトモードの far jump で使用) |
| `ks_base` | `0x2000` | GDT ベースリニアアドレス (未使用定数) |

## ラベル・ルーチンリファレンス

### start

```asm
start:
    movw  $s_base, %ax
    movw  %ax, %ds
    movw  %ax, %es
    movw  $0x1000, %ax
    movw  %ax, %ss
    movw  $1024, %sp
```

**概要:** エントリポイント。リアルモードでのセグメントレジスタ初期化と、プロテクトモード移行の準備を行う。

**処理内容:**

1. DS, ES を `s_base` (0x0300) に設定する。コード内のデータ参照が正しく解決されるようにする
2. SS を 0x1000 に設定し、SP を 1024 に設定する。スタックは 0x1000:0400 = 物理 0x10400 付近に配置される
3. `cli` で割り込みを禁止する (IDTR 変更前の安全措置)
4. `a20_init` を呼び出して A20 ラインを有効化する
5. `lidt idt_ptr` で IDTR をロードする (IDT ベースアドレス: 0x2100、リミット: 2047)
6. `lgdt gdt_ptr` で GDTR をロードする (GDT ベースアドレス: 0x2000、リミット: 255)
7. CR0 の PE ビット (ビット 0) をセットしてプロテクトモードに遷移する
8. far jump で 16 ビットコードセグメント (セレクタ 0x08) の `flush_start` へジャンプし、プリフェッチキューをフラッシュする

**呼び出し元:** boot.s (`ljmp $0x300, $0`)、SMP の SIPI (ベクタ 0x03)

### flush_start

```asm
flush_start:
    movw  $0x10, %ax
    movw  %ax, %ds
    movw  %ax, %es
    movw  $0x18, %ax
    movw  %ax, %ss
```

**概要:** プロテクトモード移行後の最初の 16 ビットコード。データセグメントとスタックセグメントをプロテクトモード用セレクタに切り替える。

**処理内容:**

1. DS, ES を 0x10 (SEL_K16_D: 16 ビットカーネルデータセグメント、ベース 0xB8000 = VGA) に設定する
2. SS を 0x18 (SEL_K16_S: 16 ビットカーネルスタックセグメント) に設定する
3. far jump でセレクタ 0x20 (SEL_K32_C: 32 ビットカーネルコードセグメント) のオフセット `k_base` (0x3400) へジャンプする。これにより run.s のエントリポイントに到達する

**注意点:** far jump はバイト列 `0xEA` + オフセット `k_base` (0x3400) + セレクタ `0x20` として直接エンコードされている。アセンブラの `ljmp` ディレクティブではなくバイト直接記述を使用しているが、オフセット値は `.equ k_base, 0x3400` で定義された定数を参照する。

### a20_init

```asm
a20_init:
    call  a20_init_wait
    movb  $0xd1, %al
    movw  $0x64, %dx
    outb  %al, %dx
    call  a20_init_wait
    movb  $0xdf, %al
    movw  $0x60, %dx
    outb  %al, %dx
    call  a20_init_wait
    ret
```

**概要:** A20 ラインを有効化する。A20 ラインが無効の場合、1MB 以上のアドレスにアクセスできない。

**処理内容:**

1. キーボードコントローラ (8042) のポート 0x64 にコマンド 0xD1 (出力ポート書き込みコマンド) を送信する
2. ポート 0x60 にデータ 0xDF を送信する。これにより A20 ラインが有効化される (0xDF = 全ビットセット、ビット 1 が A20 ゲート有効)
3. 各操作の前に `a20_init_wait` でコントローラの準備完了を待つ

**呼び出し元:** `start`

**注意点:** キーボードコントローラ経由の A20 有効化方式。Fast A20 (ポート 0x92) 方式は使用していない。

### a20_init_wait

```asm
a20_init_wait:
    movw  $0x64, %dx
    inb   %dx, %al
    andb  $0x02, %al
    jnz   a20_init_wait
    ret
```

**概要:** キーボードコントローラ (8042) の入力バッファが空になるまで待機する。

**処理内容:**

1. ステータスポート 0x64 を読み出す
2. ビット 1 (Input Buffer Full) が 0 になるまでポーリングする
3. コントローラがコマンドを受け付け可能になったらリターンする

**呼び出し元:** `a20_init`

### hello

```asm
hello:
    movb  $0x0e, %ah
    movb  $0x39, %al
    movb  $2, %bl
    int   $0x10
    ret
```

**概要:** デバッグ用ルーチン。文字 '9' (ASCII 0x39) を BIOS INT 10h で画面に表示する。

**呼び出し元:** デバッグ時に手動挿入 (通常は未使用)

### error

```asm
error:
    jmp  error
```

**概要:** エラー時の無限ループ。デバッグ用。

## データ定義

### gdt_ptr

```asm
.align  4
gdt_ptr:  .word  256-1
          .word  0x2000, 0x0
```

**概要:** GDTR にロードする GDT ディスクリプタ。

| フィールド | 値 | 説明 |
|-----------|------|------|
| リミット | 255 (256 - 1) | GDT のサイズ = 256 バイト (32 エントリ x 8 バイト) |
| ベースアドレス | 0x00002000 | GDT のリニアアドレス (table.s が配置されるアドレス) |

### idt_ptr

```asm
.align  4
idt_ptr:  .word  2047
          .word  0x2100
          .word  0x0
```

**概要:** IDTR にロードする IDT ディスクリプタ。

| フィールド | 値 | 説明 |
|-----------|------|------|
| リミット | 2047 (0x7FF) | IDT のサイズ = 2048 バイト (256 エントリ x 8 バイト) |
| ベースアドレス | 0x00002100 | IDT のリニアアドレス |

## 補足

### far jump のエンコーディング

start.s では far jump を `ljmp` ニーモニックではなく `.byte` ディレクティブで直接エンコードしている箇所が 2 つある:

```asm
# 16 ビット → プロテクトモードフラッシュ
.byte  0xea
.word  flush_start
.word  0x08          # SEL_K16_C (16 ビットカーネルコード)

# 16 ビット → 32 ビットジャンプ
.byte  0xea
.word  k_base        # 0x3400 (= -Ttext in Makefile)
.word  0x20          # SEL_K32_C (32 ビットカーネルコード)
```

これは 16 ビットモードのアセンブラが 32 ビットセレクタへの far jump を正しく生成できない場合があるための回避策である。

### プロテクトモード遷移シーケンス

1. `cli` - 割り込み禁止
2. A20 ライン有効化
3. IDTR ロード (IDT ベース: 0x2100)
4. GDTR ロード (GDT ベース: 0x2000)
5. CR0.PE = 1 (プロテクトモード有効化)
6. far jump (プリフェッチキューフラッシュ + CS ロード)
7. DS, ES, SS をプロテクトモード用セレクタに設定
8. far jump で 32 ビットコードセグメント (run.s) へ遷移

### SMP での再利用

AP (CPU 1) は SIPI により物理アドレス 0x3000 から実行を開始するため、この start.s のコードが再実行される。AP も同じ GDT/IDT を使用してプロテクトモードに遷移し、run.s で `cpu_num` を確認して AP 固有のパスに分岐する。

### .org ディレクティブ

```asm
.org  1024
```

ファイル末尾でサイズを 1024 バイトに固定している。これによりカーネルイメージ内での start.s の占有サイズが確定し、後続の run.s のオフセット (0x3400) が正しくなることを保証する。
