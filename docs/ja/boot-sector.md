## 1. ブートセクタとは

PC の電源を入れると、BIOS (Basic Input/Output System) が最初に実行される。
BIOS はハードウェアの初期化を行った後、起動デバイス (フロッピーディスクや
ハードディスク) の**最初の 512 バイト**をメモリアドレス `0x7C00` に読み込み、
そこにジャンプして実行を開始する。

この最初の 512 バイトを**ブートセクタ** (boot sector) と呼ぶ。

ブートセクタの制約:

- サイズは正確に **512 バイト** (1 セクタ)
- 末尾 2 バイトが `0x55, 0xAA` (リトルエンディアンで `0xAA55`) でなければ
  BIOS はブート可能と認識しない
- 使える命令は **16 ビットリアルモード** のみ
- OS は何もロードされていないので、ハードウェアアクセスは BIOS の
  割り込みサービス (INT) を使う

tiny-itron の `boot/boot.s` はこのブートセクタであり、フロッピーディスクから
カーネルをメモリに読み込む役割を果たす。

## 2. 前提知識: リアルモードとセグメント

### リアルモードとは

x86 CPU は電源投入直後、**リアルモード** (real mode) で動作する。
リアルモードでは:

- レジスタは **16 ビット幅** (AX, BX, CX, DX, SI, DI, SP, BP)
- 直接アドレス指定できる範囲は 16 ビット = **64KB** まで
- 1MB のメモリにアクセスするために**セグメント**を使う

### セグメント:オフセット アドレッシング

リアルモードでは、メモリアドレスを `セグメント:オフセット` の形式で指定する。
物理アドレスは次の式で計算される:

```
物理アドレス = セグメント × 16 + オフセット
```

例:

| セグメント:オフセット | 計算 | 物理アドレス |
|:---:|:---:|:---:|
| `0x07C0:0x0000` | 0x07C0 × 16 + 0x0000 | `0x7C00` |
| `0x7000:0x0000` | 0x7000 × 16 + 0x0000 | `0x70000` |
| `0x0200:0x0000` | 0x0200 × 16 + 0x0000 | `0x2000` |
| `0x0300:0x0000` | 0x0300 × 16 + 0x0000 | `0x3000` |

### セグメントレジスタ

| レジスタ | 用途 |
|:---:|:---|
| CS | コードセグメント (命令の読み出し元) |
| DS | データセグメント (データの読み書き) |
| ES | エクストラセグメント (文字列操作の転送先など) |
| SS | スタックセグメント (push/pop/call/ret で使用) |

### BIOS 割り込みサービス

リアルモードでは OS がまだ存在しないため、画面出力やディスク読み取りは
BIOS が提供する割り込みサービスを使う:

| 割り込み | 機能 |
|:---:|:---|
| `INT 0x10` | ビデオサービス (文字表示など) |
| `INT 0x13` | ディスクサービス (セクタ読み書き) |

## 3. 全体のブートフロー

tiny-itron は 3 段階でブートする:

```
  電源 ON
    |
    v
+------------------+
| BIOS             |   フロッピーの先頭 512 バイトを 0x7C00 に読み込む
+------------------+
    |
    v
+------------------+
| boot/boot.s      |   ← このドキュメントで解説するファイル
| (ブートセクタ)    |   カーネル全体をフロッピーからメモリに読み込む
| 0x7C00 → 0x70000 |   セカンドブート (start.s) にジャンプ
+------------------+
    |  ljmp $0x300, $0  (物理アドレス 0x3000)
    v
+------------------+
| start.s          |   A20 ライン有効化
| (セカンドブート)  |   GDT/IDT ロード
| 0x3000           |   プロテクトモード (32 ビット) に移行
+------------------+   run.s にジャンプ
    |  ljmp で 32 ビットセグメント 0x3400 へ
    v
+------------------+
| run.s → main.c   |   32 ビットカーネル本体の実行開始
| 0x3400           |   デバイス初期化、タスク起動
+------------------+
```

### なぜ 3 段階？

ブートセクタは 512 バイトしか使えない。この小さなプログラムで
カーネル全体をメモリに読み込むことだけに専念し、より複雑な初期化
(プロテクトモード移行) は後段の start.s に任せる。

## 4. メモリマップ

boot.s が動作する時点のメモリ配置:

```
物理アドレス       内容
============================================================
0x00000 - 0x003FF   割り込みベクタテーブル (BIOS が設定)
0x00400 - 0x004FF   BIOS データ領域
  ...
0x02000             GDT (genasm.c が生成、boot.s がロード)
0x02100             IDT
0x03000             start.s (セカンドブート、16→32ビット遷移)
0x03400             run.s → main.c (32 ビットカーネル本体)
  ...
0x07C00             ← BIOS がブートセクタをロードする場所
0x70000             ← boot.s が自分自身を再配置する先
  ...
0x7A000             CPU0 のカーネルスタック
0xA0000             VGA ビデオメモリ開始
0xB8000             VGA テキストモードビデオメモリ
============================================================
```

フロッピーディスクからの読み込み先:

```
セクタ    読み込み先              内容
=======================================================
0         0x7C00 (BIOS)          boot.s (このファイル)
1-8       0x2000                 GDT テーブル (table.s, 4096 bytes)
9-10      0x3000                 start.s (セカンドブート, 1024 bytes)
11-299    0x3400 以降             カーネル本体 (run.s + ...)
=======================================================
```

## 5. コード詳細解説

### 5.1 定数定義

```asm
.equ b_base, 0x07c0       # BIOS がブートセクタをロードするセグメント
.equ m_base, 0x7000        # 自己再配置先のセグメント
.equ kt_base, 0x0200       # カーネル読み込み先の初期セグメント
.equ s_base, 0x0300        # セカンドブート (start.s) のセグメント
.equ delta_seg, 0x1000     # 64KB 境界を超えた時のセグメント加算値
```

| 定数名 | 値 | 物理アドレス | 意味 |
|:---|:---:|:---:|:---|
| `b_base` | `0x07C0` | `0x7C00` | BIOS がロードする場所 |
| `m_base` | `0x7000` | `0x70000` | 再配置先 (安全な場所) |
| `kt_base` | `0x0200` | `0x2000` | カーネル読み込み開始点 |
| `s_base` | `0x0300` | `0x3000` | セカンドブート (start.s) の場所 |
| `delta_seg` | `0x1000` | +`0x10000` (64KB) | セグメント加算値 |

```asm
.code16        # 16 ビットコードを生成するようアセンブラに指示
.text          # テキスト (コード) セクション
.globl start   # start ラベルをリンカに公開
    .org 0     # このコードのオフセットを 0 から開始
```

### 5.2 エントリポイント (start)

BIOS がブートセクタを `0x7C00` にロードした後、ここにジャンプしてくる。

```asm
start:
    movw    $b_base, %ax       # AX = 0x07C0
    movw    %ax, %ds           # DS = 0x07C0 (データセグメント設定)
    movw    %ax, %ss           # SS = 0x07C0 (スタックセグメント設定)
    movw    $1024, %sp         # SP = 1024  (スタックポインタ設定)
```

**何をしているか:**

1. BIOS はジャンプ時に CS:IP = `0x07C0:0x0000` をセットするが、
   DS や SS は設定してくれない
2. そこで DS と SS を `0x07C0` に手動設定する
3. スタックポインタを 1024 に設定 (ブートセクタ本体の末尾付近。
   スタックは下方向に伸びるので、512〜1024 の範囲を使う)

**図解: 初期メモリ配置**
```
0x7C00  +-------------------+  ← DS:0 = SS:0 の指す場所
        | boot.s 本体       |
        | (512 バイト)       |
0x7E00  +-------------------+  ← ブートセクタの終端
        | スタック領域       |
        | (下方向に伸びる)   |
0x8000  +-------------------+  ← SS:SP=1024 = SS:0x400
```

### 5.3 自己再配置

ブートセクタはこれからフロッピーからカーネルをメモリに読み込むが、
読み込み先が `0x2000` 付近で、自分自身の `0x7C00` と干渉する可能性がある。
安全のため、自分自身を別のアドレス (`0x70000`) にコピーして、そちらで
実行を続ける。

```asm
    # 自分自身を 0x7000:0000 (= 0x70000) にコピー
    movw    $m_base, %ax       # AX = 0x7000
    movw    %ax, %es           # ES = 0x7000 (コピー先セグメント)
    movw    $0, %di            # DI = 0 (コピー先オフセット)
    movw    $start, %si        # SI = 0 (コピー元オフセット = start ラベル)
    movw    $512, %cx          # CX = 512 (コピーするバイト数)
    rep     movsb              # DS:SI → ES:DI を CX 回繰り返す
```

**`rep movsb` の動作:**

`rep movsb` は x86 の文字列操作命令で、次を CX 回繰り返す:
1. DS:SI のバイトを ES:DI にコピー
2. SI と DI を 1 ずつ増やす
3. CX を 1 減らす

結果: 512 バイトが `0x07C0:0x0000` から `0x7000:0x0000` にコピーされる。

```asm
    ljmp    $m_base, $jumphere   # 0x7000:jumphere にジャンプ
```

**`ljmp` (far jump):**

CS レジスタも含めてジャンプする命令。これにより実行が
コピー先のコードに移る。`ljmp` を使わないと CS が古い値
(`0x07C0`) のままで、コピー前のコードを実行し続けてしまう。

```asm
jumphere:
    movw    $m_base, %ax
    movw    %ax, %ds           # DS = 0x7000
    movw    %ax, %es           # ES = 0x7000
    movw    %ax, %ss           # SS = 0x7000
    movw    $1024, %sp         # SP = 1024
```

再配置先でセグメントレジスタを再設定。以降、すべてのデータアクセスは
`0x7000` セグメント基準になる。

### 5.4 メッセージ表示とフロッピー初期化

```asm
    movw    $message, %bx      # BX = message 文字列のオフセット
    pushw   %bx                # 引数をスタックに積む
    call    puts               # 文字列を画面に表示
    popw    %bx                # スタックを掃除
    call    init_fd            # フロッピーディスクコントローラ初期化
```

画面に著作権表示メッセージを出力した後、フロッピーディスクの
パラメータテーブルを初期化する。

**`init_fd` の仕組み:**

```asm
init_fd:
    pushw   %es
    movw    $0x78, %bx         # 0x0000:0x0078 = BIOS のフロッピーパラメータへの
    movw    $0, %ax            #   ポインタが格納されている場所
    movw    %ax, %es
    movw    $fd144, %es:(%bx)  # ポインタのオフセットを fd144 に変更
    movw    %ds, %es:2(%bx)    # ポインタのセグメントを DS に変更
    popw    %es
    ret
```

BIOS は物理アドレス `0x0000:0x0078` にフロッピーディスクパラメータテーブルへの
ポインタを保持している。`init_fd` はこのポインタを、boot.s 内に定義した
`fd144` テーブルに差し替える。`fd144` は 1.44MB フロッピーの仕様に
合わせたパラメータを含む:

```asm
fd144:  .byte 0xaf, 2, 25, 2, 36, 0x1b, 0xff, 0x54, 0xf6, 15, 8
```

特に重要なのは **36** (シリンダあたりのセクタ数 = 18 セクタ/トラック × 2 ヘッド)。
`divb` 命令で論理セクタ番号をこの値で除算し、商がシリンダ番号、
剰余がヘッド・セクタに分解される。

### 5.5 カーネル読み込み (load)

ブートセクタのメインの仕事: フロッピーからカーネルをメモリに読み込む。

```asm
load:
    movw    $kt_base, %ax     # AX = 0x0200
    movw    %ax, %es          # ES = 0x0200 (読み込み先セグメント = 物理 0x2000)
    movw    $0x1, %ax          # AX = 1 (論理セクタ番号。0 はブートセクタ自身)
    movw    $0x0000, %bx       # BX = 0 (セグメント内オフセット)
```

**論理セクタの考え方:**

フロッピーディスクは物理的にはシリンダ・ヘッド・セクタ (CHS) で
アドレスされるが、プログラム上は連続した番号 (論理セクタ) で扱う方が簡単:

```
論理セクタ 0      = ブートセクタ (boot.s, 512 bytes)
論理セクタ 1〜8   = GDT テーブル (table.s, 4096 bytes)
論理セクタ 9〜10  = start.s (セカンドブート, 1024 bytes)
論理セクタ 11〜   = カーネル本体 (run.s → main.c)
 ...
論理セクタ 299    = カーネルの末尾
```

**読み込みループ:**

```asm
load_1:
    pushw   %bx                # ES:BX = 読み込み先アドレスを保存
    pushw   %ax                # AX = 論理セクタ番号を保存
    call    readsect           # 1 セクタ (512 バイト) を読み込む
    popw    %ax
    popw    %bx
    addw    $512, %bx          # オフセットを 512 バイト進める
    incw    %ax                # 次のセクタ番号へ
```

1 回のループで 1 セクタ (512 バイト) を読み込み、読み込み先を 512 バイト
進める。これを論理セクタ 299 まで繰り返す。

合計読み込みサイズ: セクタ 1 から 299 まで (AX が 300 に達したら終了) = 299 セクタ。
299 × 512 = **153,088 バイト** (約 149.5 KB)

### 5.6 セクタ読み取り (readsect)

論理セクタ番号を CHS (Cylinder, Head, Sector) に変換して BIOS INT 0x13 で
ディスクから読み取る関数。

**1.44MB フロッピーの仕様:**

| パラメータ | 値 |
|:---|:---:|
| シリンダ (トラック) 数 | 80 (0〜79) |
| ヘッド数 | 2 (0〜1) |
| 1 トラックあたりのセクタ数 | 18 (1〜18) |
| 1 セクタのサイズ | 512 バイト |
| 合計容量 | 80 × 2 × 18 × 512 = 1,474,560 バイト |

**論理セクタ → CHS 変換 (2 段階の除算):**

```
前提: 1 シリンダ = 2 ヘッド × 18 セクタ = 36 セクタ

【第 1 段階】論理セクタ ÷ 36
    商  → シリンダ番号
    余り → そのシリンダ内での位置 (0〜35)

【第 2 段階】上の余り ÷ 18
    商  → ヘッド番号 (0 または 1)
    余り → セクタ番号 (0〜17) → BIOS は 1 始まりなので +1 する

例: 論理セクタ 55
    55 ÷ 36 = 商 1, 余り 19   → シリンダ 1
    19 ÷ 18 = 商 1, 余り 1    → ヘッド 1, セクタ 1+1 = 2
    → CHS = (1, 1, 2)
```

**コードの対応:**

```asm
readsect:
    pushw   %bp
    movw    %sp, %bp

    movw    4(%bp), %ax        # AX = 論理セクタ番号 (引数)
    movb    $36, %dl
    divb    %dl                # AL = AX ÷ 36 の商 (シリンダ)
                               # AH = AX ÷ 36 の余り (ヘッド × 18 + セクタ)
    movb    %al, %ch           # CH = シリンダ番号
    movb    %ah, %al           # AL = 余り
    movb    $0, %ah

    movb    $18, %dl
    divb    %dl                # AL = 余り ÷ 18 の商 (ヘッド)
                               # AH = 余り ÷ 18 の余り (セクタ - 1)
    movb    %al, %dh           # DH = ヘッド番号
    movb    %ah, %cl           # CL = セクタ番号 (0 始まり)

    incb    %cl                # CL += 1 (BIOS はセクタ番号を 1 始まりで数える)
```

**`divb` 命令の動作:**

`divb %dl` は AX を DL で割る:
- 商 → AL
- 余り → AH

これは 8 ビット除算なので、AX (16 ビット) ÷ DL (8 ビット) で計算される。

**BIOS INT 0x13 呼び出し:**

```asm
    movb    $0, %dl            # DL = ドライブ番号 (0 = 最初のフロッピー)
    movb    $2, %ah            # AH = 0x02 (セクタ読み取り機能)
    movb    $1, %al            # AL = 読み取りセクタ数 (1 セクタ)
    int     $0x13              # BIOS ディスクサービス呼び出し
```

INT 0x13, AH=02h のパラメータ:

| レジスタ | 意味 |
|:---:|:---|
| AH | `0x02` = セクタ読み取り |
| AL | 読み取るセクタ数 |
| CH | シリンダ番号 |
| CL | セクタ番号 (1 始まり) |
| DH | ヘッド番号 |
| DL | ドライブ番号 (0 = A ドライブ) |
| ES:BX | 読み込み先メモリアドレス |

成功すると CF (キャリーフラグ) = 0、失敗すると CF = 1。

**進捗表示 (スピナーアニメーション):**

読み取り後、`-\|/` の回転アニメーションを表示して進捗を示す:

```asm
    movw    dot, %ax           # カウンタを読み込む
    incw    %ax                # インクリメント
    movw    %ax, dot
    shrw    $2, %ax            # 4 回に 1 回パターンを変更
    andw    $0x0003, %ax       # 0, 1, 2, 3 のいずれかを得る
    movw    $dot_data, %bx     # "-\|/" テーブルのアドレス
    addw    %ax, %bx           # テーブル内のインデックス
    movb    (%bx), %al         # 対応する文字を取得
    movb    %al, dot1          # 表示文字列に書き込む
    movw    $dot0, %bx         # 表示文字列 = BS + 文字 + NUL
```

`dot0` は `0x08` (バックスペース) で始まる文字列。前の文字を上書きして、
`-`, `\`, `|`, `/` が順に表示され、回転しているように見える:

```
Loading... -    (最初)
Loading... \    (4 セクタ後)
Loading... |    (8 セクタ後)
Loading... /    (12 セクタ後)
Loading... -    (16 セクタ後、一巡)
```

エラー時は `jnc readsect_1` が不成立となり、"Read Sector Error" メッセージを
表示する。

### 5.7 セグメントローテーション

16 ビットリアルモードでは、オフセットレジスタ (BX) は 16 ビットなので
`0x0000`〜`0xFFFF` (64KB) の範囲しか指定できない。カーネルが 64KB を超える場合、
BX がオーバーフローして 0 に戻ってしまう。

```asm
    testw   $0xffff, %bx       # BX がオーバーフローして 0 になったか？
    jnz     over_1             # 0 でなければスキップ

    # BX が 0 に戻った = 64KB 境界を越えた
    pushw   %ax
    movw    %es, %ax           # 現在の ES を取得
    addw    $delta_seg, %ax    # ES += 0x1000 (= 物理アドレスで +64KB)
    movw    %ax, %es           # 新しい ES をセット
    popw    %ax
over_1:
```

**なぜ `$0x1000` を足すのか:**

ES に `0x1000` を足すと、物理アドレスでは `0x1000 × 16 = 0x10000` (64KB) 増える。
BX が 0 に戻った分を ES の増分で補う。

**図解:**

```
最初:   ES = 0x0200, BX = 0x0000  → 物理アドレス 0x2000
        BX が 0xFFFF まで増える...

回転後: ES = 0x1200, BX = 0x0000  → 物理アドレス 0x12000
        (0x200 + 0x1000 = 0x1200)
        BX が再び増えていく...

次の回転: ES = 0x2200, BX = 0x0000 → 物理アドレス 0x22000
```

### 5.8 読み込み完了とセカンドブートへのジャンプ

```asm
    cmpw    $300, %ax          # 論理セクタ 300 まで読んだ？
    je      load_2             # はい → 読み込み完了
    jmp     load_1             # いいえ → 次のセクタを読む

load_2:
    ret                        # load 関数から戻る
```

load 関数から戻ると:

```asm
    movw    $secondboot, %bx   # "Second Boot ..." メッセージ
    pushw   %bx
    call    puts               # 画面に表示
    popw    %bx
    ljmp    $0x300, $0         # 0x0300:0x0000 (= 物理 0x3000) にジャンプ
```

`ljmp $0x300, $0` でセカンドブートローダ (start.s) に制御を移す。
start.s は物理アドレス `0x3000` にロードされており、ここから
プロテクトモード (32 ビット) への移行が始まる。

```asm
loop0:
    jmp     loop0              # 万が一戻ってきた場合の無限ループ (安全策)
```

## 6. ユーティリティ関数

### puts - 文字列表示

```asm
puts:
    pushw   %bp
    movw    %sp, %bp           # スタックフレーム設定
    pushw   %bx
    pushw   %cx

    movw    4(%bp), %bx        # 引数: 文字列のアドレス
puts_1:
    movb    (%bx), %al         # 1 文字読み込む
    cmpb    $0x00, %al         # NUL 終端？
    je      puts_2             # はい → 終了
    call    putc               # 1 文字表示
    incw    %bx                # 次の文字へ
    jmp     puts_1
puts_2:
    popw    %cx
    popw    %bx
    popw    %bp
    ret
```

C 言語の `while (*s) putchar(*s++);` に相当する。引数はスタック経由で渡される
(16 ビットの呼び出し規約)。

### putc - 1 文字表示

```asm
putc:
    pushw   %bx
    movb    $0x0e, %ah         # AH = 0x0E (テレタイプ出力)
    movb    $2, %bl            # BL = 2 (表示色: 緑)
    int     $0x10              # BIOS ビデオサービス
    popw    %bx
    ret
```

BIOS INT 0x10, AH=0x0E (テレタイプ出力) を使って 1 文字を画面に表示する。
テレタイプモードでは改行 (`0x0A`) やキャリッジリターン (`0x0D`) も正しく処理される。

## 8. ブートシグネチャ

```asm
    .org    510                # オフセット 510 バイト目に移動
    .word   0xaa55             # ブートシグネチャ
```

ブートセクタの最後の 2 バイト (オフセット 510-511) に `0x55, 0xAA` を配置する。
リトルエンディアンなので `.word 0xaa55` は `0x55, 0xAA` の順にメモリに格納される。

BIOS はこのシグネチャを見てブート可能なディスクかどうかを判断する。
この 2 バイトがなければ BIOS はブートを拒否する。

`.org 510` は「ここまでのコード+データが 510 バイト未満なら、残りを 0 で
パディングする」指示。つまりブートセクタ全体が正確に 512 バイトになる。

```
オフセット  内容
==========================================
0x000       start: (エントリポイント)
  ...       コード + データ
  ...       (0 パディング)
0x1FE       0x55  ← ブートシグネチャ
0x1FF       0xAA
==========================================
合計: 512 バイト (1 セクタ)
```

## 9. ビルド方法

```bash
cd i386/boot
make
```

Makefile の内容:

```makefile
boot: boot.s
    as --32 -o boot.o boot.s        # 32 ビットモードでアセンブル
    ld -m elf_i386 --oformat binary -Ttext 0x0 -o boot boot.o
    rm -f boot.o
```

| オプション | 意味 |
|:---|:---|
| `as --32` | 32 ビット ELF を出力 (`.code16` で 16 ビットコードを生成) |
| `--oformat binary` | ELF ヘッダなしの生バイナリを出力 |
| `-Ttext 0x0` | テキストセクションの開始アドレスを 0 に設定 |

生成された `boot` (512 バイト) は、最終的にフロッピーイメージの先頭に
結合される:

```makefile
# i386/Makefile の最終行
cat boot/boot table start kernel > i386
```

## 10. フロッピーディスクイメージの構造

### 10.1 イメージの構成

Makefile の最終ステップで、4 つのバイナリファイルを `cat` で単純に連結して
1 つのフロッピーイメージを作る:

```makefile
# i386/Makefile
cat boot/boot table start kernel > i386
```

```
ファイル     内容              サイズ        イメージ内オフセット
=====================================================================
boot/boot    ブートセクタ      512 バイト     0x0000 (セクタ 0)
table        GDT テーブル      4096 バイト    0x0200 (セクタ 1〜8)
start        セカンドブート    1024 バイト    0x1200 (セクタ 9〜10)
kernel       カーネル本体      約 107 KB      0x1600 (セクタ 11〜)
=====================================================================
```

### 10.2 なぜサイズが固定されているか — `.org` ディレクティブ

`cat` による連結では、各ファイルのサイズが 1 バイトでもずれると後続の
ファイルの配置がすべてずれてしまう。そのため boot, table, start の 3 つは
**`.org` ディレクティブでサイズを明示的に固定** している:

| ファイル | `.org` の値 | サイズ | ソース |
|----------|------------|--------|--------|
| boot/boot | `.org 510` + `.word 0xAA55` | 512 バイト | `boot/boot.s:454-455` |
| table | `.org 4096` | 4096 バイト | `genasm.c:72` が生成する `table.s` |
| start | `.org 1024` | 1024 バイト | `start.s:253` |
| kernel | (なし — 自然サイズ) | 約 107 KB | リンカ出力を `elf` ツールで変換 |

`.org N` は出力バイナリのサイズを N バイトに固定する (実際のコードが N バイト
未満なら、残りはゼロで埋められる)。これにより、各ファイルの実体がどれだけ
小さくても、`cat` 連結後のオフセットは常に一定になる。

### 10.3 ブートローダによるメモリ配置

BIOS がフロッピーのセクタ 0 (= boot/boot) を `0x7C00` に読み込んで実行する。
boot.s はセクタ 1〜299 をセグメント `0x0200` (リニアアドレス `0x2000`) 以降に
順番に読み込む。すると連結された各ファイルは以下のリニアアドレスに配置される:

```
イメージ内         読み込み先           計算
オフセット         リニアアドレス
─────────────────────────────────────────────────────────
0x0000 (boot)      0x7C00              BIOS がロード (セクタ 0)
0x0200 (table)     0x2000              0x2000 + 0x0000 = 0x2000
0x1200 (start)     0x3000              0x2000 + 0x1000 = 0x3000
0x1600 (kernel)    0x3400              0x2000 + 0x1400 = 0x3400
─────────────────────────────────────────────────────────
```

ここでの計算:
- table (4096 バイト = 0x1000) が 0x2000 に配置される
- start は table の直後なので 0x2000 + 0x1000 = **0x3000**
- kernel は start の直後なので 0x3000 + 0x0400 (1024) = **0x3400**

### 10.4 リンカの `-Ttext=0x3400` との対応

カーネルのリンカオプションは以下の通り:

```makefile
LDFLAGS = -m elf_i386 -n -T kernel.ld -static
```

`kernel.ld` 内の `. = 0x3400` が「カーネルのコードはリニアアドレス 0x3400 に
配置される」ことを指定し、リンカはこれを前提にアドレスを解決する。ブートローダが実際にカーネルを 0x3400 に
読み込むので、リンカの想定と実行時の配置が一致する。

**つまり、この仕組み全体は次の連鎖で成り立っている:**

1. `.org` で boot (512B), table (4096B), start (1024B) のサイズを固定
2. `cat` で連結 → イメージ内のオフセットが確定
3. boot.s がセクタ 1 以降を 0x2000 に読み込む → 各ファイルのリニアアドレスが確定
4. table → 0x2000 (= `AL_GDT`)、start → 0x3000 (= `AL_KERNEL16`)、
   kernel → 0x3400 (= `-Ttext=0x3400`) と、すべてが整合する

### 10.5 QEMU での実行

QEMU での実行時、このイメージは 1.44MB フロッピーに書き込まれる
(`run.sh` が `dd` で作成):

```bash
dd if=/dev/zero of=floppy.img bs=512 count=2880    # 1.44MB の空イメージ
dd if=i386 of=floppy.img conv=notrunc              # 先頭にカーネルイメージを書き込む
```
