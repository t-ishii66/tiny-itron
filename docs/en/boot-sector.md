## 1. What Is a Boot Sector?

When a PC is powered on, the BIOS (Basic Input/Output System) executes first.
After initializing the hardware, the BIOS reads the **first 512 bytes** from the
boot device (floppy disk or hard disk) into memory address `0x7C00` and jumps
there to begin execution.

These first 512 bytes are called the **boot sector**.

Constraints of the boot sector:

- Size is exactly **512 bytes** (1 sector)
- The last 2 bytes must be `0x55, 0xAA` (i.e., `0xAA55` in little-endian),
  otherwise the BIOS will not recognize the disk as bootable
- Only **16-bit real mode** instructions are available
- Since no OS has been loaded yet, hardware access relies on BIOS
  interrupt services (INT)

tiny-itron's `boot/boot.s` is this boot sector, and its job is to load the
kernel from the floppy disk into memory.

## 2. Background: Real Mode and Segments

### What Is Real Mode?

Immediately after power-on, the x86 CPU operates in **real mode**.
In real mode:

- Registers are **16 bits wide** (AX, BX, CX, DX, SI, DI, SP, BP)
- The directly addressable range is limited to 16 bits = **64KB**
- **Segments** are used to access the full 1MB of memory

### Segment:Offset Addressing

In real mode, memory addresses are specified in the form `segment:offset`.
The physical address is calculated as follows:

```
physical address = segment × 16 + offset
```

Examples:

| Segment:Offset | Calculation | Physical Address |
|:---:|:---:|:---:|
| `0x07C0:0x0000` | 0x07C0 × 16 + 0x0000 | `0x7C00` |
| `0x7000:0x0000` | 0x7000 × 16 + 0x0000 | `0x70000` |
| `0x0200:0x0000` | 0x0200 × 16 + 0x0000 | `0x2000` |
| `0x0300:0x0000` | 0x0300 × 16 + 0x0000 | `0x3000` |

### Segment Registers

| Register | Purpose |
|:---:|:---|
| CS | Code segment (where instructions are fetched from) |
| DS | Data segment (for data reads and writes) |
| ES | Extra segment (destination for string operations, etc.) |
| SS | Stack segment (used by push/pop/call/ret) |

### BIOS Interrupt Services

In real mode, since no OS exists yet, screen output and disk reads use
interrupt services provided by the BIOS:

| Interrupt | Function |
|:---:|:---|
| `INT 0x10` | Video services (character display, etc.) |
| `INT 0x13` | Disk services (sector read/write) |

## 3. Overall Boot Flow

tiny-itron boots in 3 stages:

```
  Power ON
    |
    v
+------------------+
| BIOS             |   Reads the first 512 bytes of the floppy to 0x7C00
+------------------+
    |
    v
+------------------+
| boot/boot.s      |   <-- The file explained in this document
| (boot sector)    |   Loads the entire kernel from floppy into memory
| 0x7C00 -> 0x70000|   Jumps to the second-stage boot (start.s)
+------------------+
    |  ljmp $0x300, $0  (physical address 0x3000)
    v
+------------------+
| start.s          |   Enables A20 line
| (second-stage    |   Loads GDT/IDT
|  boot)           |   Transitions to protected mode (32-bit)
| 0x3000           |   Jumps to run.s
+------------------+
    |  ljmp to 32-bit segment 0x3400
    v
+------------------+
| run.s -> main.c  |   Starts executing the 32-bit kernel
| 0x3400           |   Device initialization, task startup
+------------------+
```

### Why 3 Stages?

The boot sector can only use 512 bytes. This tiny program focuses solely on
loading the entire kernel into memory, and delegates the more complex
initialization (such as transitioning to protected mode) to the subsequent
start.s.

## 5. Code Walkthrough

### 5.1 Constant Definitions

```asm
.equ b_base, 0x07c0       # Segment where BIOS loads the boot sector
.equ m_base, 0x7000        # Segment for self-relocation destination
.equ kt_base, 0x0200       # Initial segment for kernel load destination
.equ s_base, 0x0300        # Segment for second-stage boot (start.s)
.equ delta_seg, 0x1000     # Segment increment when crossing a 64KB boundary
```

| Constant | Value | Physical Address | Meaning |
|:---|:---:|:---:|:---|
| `b_base` | `0x07C0` | `0x7C00` | Where BIOS loads the boot sector |
| `m_base` | `0x7000` | `0x70000` | Relocation destination (safe location) |
| `kt_base` | `0x0200` | `0x2000` | Kernel load start point |
| `s_base` | `0x0300` | `0x3000` | Second-stage boot (start.s) location |
| `delta_seg` | `0x1000` | +`0x10000` (64KB) | Segment increment value |

```asm
.code16        # Tell the assembler to generate 16-bit code
.text          # Text (code) section
.globl start   # Export the start label to the linker
    .org 0     # Start this code at offset 0
```

### 5.2 Entry Point (start)

After the BIOS loads the boot sector at `0x7C00`, it jumps here.

```asm
start:
    movw    $b_base, %ax       # AX = 0x07C0
    movw    %ax, %ds           # DS = 0x07C0 (set data segment)
    movw    %ax, %ss           # SS = 0x07C0 (set stack segment)
    movw    $1024, %sp         # SP = 1024  (set stack pointer)
```

**What this does:**

1. The BIOS sets CS:IP = `0x07C0:0x0000` when it jumps here, but
   it does not set DS or SS
2. So we manually set DS and SS to `0x07C0`
3. The stack pointer is set to 1024 (near the end of the boot sector area.
   Since the stack grows downward, it uses the range from 512 to 1024)

**Diagram: Initial memory layout**
```
0x7C00  +-------------------+  <-- Where DS:0 = SS:0 points
        | boot.s body       |
        | (512 bytes)       |
0x7E00  +-------------------+  <-- End of boot sector
        | Stack area        |
        | (grows downward)  |
0x8000  +-------------------+  <-- SS:SP=1024 = SS:0x400
```

### 5.3 Self-Relocation

The boot sector is about to load the kernel from the floppy into memory,
but the load destination is around `0x2000`, which could potentially
conflict with its own location at `0x7C00`. As a safety measure, it copies
itself to a different address (`0x70000`) and continues execution there.

```asm
    # Copy ourselves to 0x7000:0000 (= 0x70000)
    movw    $m_base, %ax       # AX = 0x7000
    movw    %ax, %es           # ES = 0x7000 (destination segment)
    movw    $0, %di            # DI = 0 (destination offset)
    movw    $start, %si        # SI = 0 (source offset = start label)
    movw    $512, %cx          # CX = 512 (number of bytes to copy)
    rep     movsb              # Repeat DS:SI -> ES:DI, CX times
```

**How `rep movsb` works:**

`rep movsb` is an x86 string operation instruction that repeats the following CX times:
1. Copy the byte at DS:SI to ES:DI
2. Increment SI and DI by 1
3. Decrement CX by 1

Result: 512 bytes are copied from `0x07C0:0x0000` to `0x7000:0x0000`.

```asm
    ljmp    $m_base, $jumphere   # Jump to 0x7000:jumphere
```

**`ljmp` (far jump):**

This instruction jumps while also updating the CS register. This transfers
execution to the copied code. Without `ljmp`, CS would remain at its old
value (`0x07C0`), and execution would continue from the pre-copy code.

```asm
jumphere:
    movw    $m_base, %ax
    movw    %ax, %ds           # DS = 0x7000
    movw    %ax, %es           # ES = 0x7000
    movw    %ax, %ss           # SS = 0x7000
    movw    $1024, %sp         # SP = 1024
```

Segment registers are reconfigured at the relocation destination. From this
point on, all data accesses use the `0x7000` segment as their base.

### 5.4 Message Display and Floppy Initialization

```asm
    movw    $message, %bx      # BX = offset of the message string
    pushw   %bx                # Push argument onto the stack
    call    puts               # Display the string on screen
    popw    %bx                # Clean up the stack
    call    init_fd            # Initialize the floppy disk controller
```

After printing a copyright message to the screen, the floppy disk
parameter table is initialized.

**How `init_fd` works:**

```asm
init_fd:
    pushw   %es
    movw    $0x78, %bx         # 0x0000:0x0078 = location where BIOS stores the
    movw    $0, %ax            #   pointer to floppy parameters
    movw    %ax, %es
    movw    $fd144, %es:(%bx)  # Replace the pointer's offset with fd144
    movw    %ds, %es:2(%bx)    # Replace the pointer's segment with DS
    popw    %es
    ret
```

The BIOS maintains a pointer to the floppy disk parameter table at physical
address `0x0000:0x0078`. `init_fd` replaces this pointer with the `fd144`
table defined within boot.s. `fd144` contains parameters tailored to the
1.44MB floppy specification:

```asm
fd144:  .byte 0xaf, 2, 25, 2, 36, 0x1b, 0xff, 0x54, 0xf6, 15, 8
```

The particularly important value here is **36** (sectors per cylinder =
18 sectors/track x 2 heads). The `divb` instruction divides the logical
sector number by this value: the quotient gives the cylinder number, and
the remainder is decomposed into head and sector.

### 5.5 Kernel Loading (load)

The main job of the boot sector: loading the kernel from the floppy into memory.

```asm
load:
    movw    $kt_base, %ax     # AX = 0x0200
    movw    %ax, %es          # ES = 0x0200 (load destination segment = physical 0x2000)
    movw    $0x1, %ax          # AX = 1 (logical sector number; 0 is the boot sector itself)
    movw    $0x0000, %bx       # BX = 0 (offset within segment)
```

**The concept of logical sectors:**

Floppy disks are physically addressed by Cylinder, Head, and Sector (CHS),
but it is easier to handle them programmatically as consecutive numbers
(logical sectors):

```
Logical sector 0      = Boot sector (boot.s, 512 bytes)
Logical sectors 1-8   = GDT table (table.s, 4096 bytes)
Logical sectors 9-10  = start.s (second-stage boot, 1024 bytes)
Logical sectors 11+   = Kernel body (run.s -> main.c)
 ...
Logical sector 299    = End of kernel
```

**The load loop:**

```asm
load_1:
    pushw   %bx                # Save ES:BX = load destination address
    pushw   %ax                # Save AX = logical sector number
    call    readsect           # Read 1 sector (512 bytes)
    popw    %ax
    popw    %bx
    addw    $512, %bx          # Advance offset by 512 bytes
    incw    %ax                # Move to the next sector number
```

Each iteration of the loop reads 1 sector (512 bytes) and advances the
destination by 512 bytes. This repeats until logical sector 299.

Total data loaded: sectors 1 through 299 (the loop exits when AX reaches 300)
= 299 sectors. 299 x 512 = **153,088 bytes** (approximately 149.5 KB)

### 5.6 Sector Read (readsect)

A function that converts a logical sector number to CHS (Cylinder, Head,
Sector) and reads from disk using BIOS INT 0x13.

**1.44MB floppy specifications:**

| Parameter | Value |
|:---|:---:|
| Number of cylinders (tracks) | 80 (0-79) |
| Number of heads | 2 (0-1) |
| Sectors per track | 18 (1-18) |
| Sector size | 512 bytes |
| Total capacity | 80 x 2 x 18 x 512 = 1,474,560 bytes |

**Logical sector to CHS conversion (two-stage division):**

```
Premise: 1 cylinder = 2 heads x 18 sectors = 36 sectors

[Stage 1] logical sector / 36
    quotient  -> cylinder number
    remainder -> position within that cylinder (0-35)

[Stage 2] above remainder / 18
    quotient  -> head number (0 or 1)
    remainder -> sector number (0-17) -> BIOS counts from 1, so add +1

Example: logical sector 55
    55 / 36 = quotient 1, remainder 19   -> cylinder 1
    19 / 18 = quotient 1, remainder 1    -> head 1, sector 1+1 = 2
    -> CHS = (1, 1, 2)
```

**Corresponding code:**

```asm
readsect:
    pushw   %bp
    movw    %sp, %bp

    movw    4(%bp), %ax        # AX = logical sector number (argument)
    movb    $36, %dl
    divb    %dl                # AL = AX / 36 quotient (cylinder)
                               # AH = AX / 36 remainder (head * 18 + sector)
    movb    %al, %ch           # CH = cylinder number
    movb    %ah, %al           # AL = remainder
    movb    $0, %ah

    movb    $18, %dl
    divb    %dl                # AL = remainder / 18 quotient (head)
                               # AH = remainder / 18 remainder (sector - 1)
    movb    %al, %dh           # DH = head number
    movb    %ah, %cl           # CL = sector number (0-based)

    incb    %cl                # CL += 1 (BIOS counts sector numbers starting from 1)
```

**How the `divb` instruction works:**

`divb %dl` divides AX by DL:
- Quotient -> AL
- Remainder -> AH

This is an 8-bit division, so the calculation is AX (16-bit) / DL (8-bit).

**BIOS INT 0x13 call:**

```asm
    movb    $0, %dl            # DL = drive number (0 = first floppy)
    movb    $2, %ah            # AH = 0x02 (read sectors function)
    movb    $1, %al            # AL = number of sectors to read (1 sector)
    int     $0x13              # Call BIOS disk service
```

INT 0x13, AH=02h parameters:

| Register | Meaning |
|:---:|:---|
| AH | `0x02` = read sectors |
| AL | Number of sectors to read |
| CH | Cylinder number |
| CL | Sector number (1-based) |
| DH | Head number |
| DL | Drive number (0 = A drive) |
| ES:BX | Destination memory address |

On success, CF (carry flag) = 0; on failure, CF = 1.

**Progress display (spinner animation):**

After each read, a rotating animation of `-\|/` is displayed to show progress:

```asm
    movw    dot, %ax           # Load the counter
    incw    %ax                # Increment
    movw    %ax, dot
    shrw    $2, %ax            # Change the pattern every 4 reads
    andw    $0x0003, %ax       # Get one of 0, 1, 2, 3
    movw    $dot_data, %bx     # Address of the "-\|/" table
    addw    %ax, %bx           # Index into the table
    movb    (%bx), %al         # Get the corresponding character
    movb    %al, dot1          # Write to the display string
    movw    $dot0, %bx         # Display string = BS + character + NUL
```

`dot0` is a string that starts with `0x08` (backspace). It overwrites the
previous character so that `-`, `\`, `|`, `/` are displayed in sequence,
creating a spinning effect:

```
Loading... -    (initially)
Loading... \    (after 4 sectors)
Loading... |    (after 8 sectors)
Loading... /    (after 12 sectors)
Loading... -    (after 16 sectors, full cycle)
```

On error, the `jnc readsect_1` branch is not taken, and a "Read Sector Error"
message is displayed.

### 5.7 Segment Rotation

In 16-bit real mode, the offset register (BX) is 16 bits, so it can only
specify the range `0x0000` to `0xFFFF` (64KB). If the kernel exceeds 64KB,
BX will overflow and wrap around to 0.

```asm
    testw   $0xffff, %bx       # Has BX overflowed and wrapped to 0?
    jnz     over_1             # If not 0, skip

    # BX wrapped to 0 = crossed a 64KB boundary
    pushw   %ax
    movw    %es, %ax           # Get current ES
    addw    $delta_seg, %ax    # ES += 0x1000 (= +64KB in physical address)
    movw    %ax, %es           # Set new ES
    popw    %ax
over_1:
```

**Why add `$0x1000`:**

Adding `0x1000` to ES increases the physical address by `0x1000 x 16 = 0x10000`
(64KB). This compensates for BX wrapping back to 0.

**Diagram:**

```
Initially: ES = 0x0200, BX = 0x0000  -> physical address 0x2000
           BX increases up to 0xFFFF...

After rotation: ES = 0x1200, BX = 0x0000  -> physical address 0x12000
                (0x200 + 0x1000 = 0x1200)
                BX starts increasing again...

Next rotation: ES = 0x2200, BX = 0x0000 -> physical address 0x22000
```

### 5.8 Load Complete and Jump to Second-Stage Boot

```asm
    cmpw    $300, %ax          # Have we read up to logical sector 300?
    je      load_2             # Yes -> loading complete
    jmp     load_1             # No -> read the next sector

load_2:
    ret                        # Return from the load function
```

After returning from the load function:

```asm
    movw    $secondboot, %bx   # "Second Boot ..." message
    pushw   %bx
    call    puts               # Display on screen
    popw    %bx
    ljmp    $0x300, $0         # Jump to 0x0300:0x0000 (= physical 0x3000)
```

`ljmp $0x300, $0` transfers control to the second-stage bootloader (start.s).
start.s has been loaded at physical address `0x3000`, and from here the
transition to protected mode (32-bit) begins.

```asm
loop0:
    jmp     loop0              # Infinite loop in case we somehow return (safety measure)
```

## 6. Utility Functions

### puts - String Display

```asm
puts:
    pushw   %bp
    movw    %sp, %bp           # Set up stack frame
    pushw   %bx
    pushw   %cx

    movw    4(%bp), %bx        # Argument: address of the string
puts_1:
    movb    (%bx), %al         # Read 1 character
    cmpb    $0x00, %al         # NUL terminator?
    je      puts_2             # Yes -> done
    call    putc               # Display 1 character
    incw    %bx                # Move to the next character
    jmp     puts_1
puts_2:
    popw    %cx
    popw    %bx
    popw    %bp
    ret
```

This is equivalent to C's `while (*s) putchar(*s++);`. The argument is passed
via the stack (16-bit calling convention).

### putc - Single Character Display

```asm
putc:
    pushw   %bx
    movb    $0x0e, %ah         # AH = 0x0E (teletype output)
    movb    $2, %bl            # BL = 2 (display color: green)
    int     $0x10              # BIOS video service
    popw    %bx
    ret
```

This uses BIOS INT 0x10, AH=0x0E (teletype output) to display a single
character on screen. In teletype mode, newline (`0x0A`) and carriage return
(`0x0D`) are processed correctly.

## 8. Boot Signature

```asm
    .org    510                # Move to byte offset 510
    .word   0xaa55             # Boot signature
```

The last 2 bytes of the boot sector (offset 510-511) are set to `0x55, 0xAA`.
Since x86 is little-endian, `.word 0xaa55` stores `0x55, 0xAA` in that order
in memory.

The BIOS checks for this signature to determine whether the disk is bootable.
Without these 2 bytes, the BIOS will refuse to boot.

`.org 510` means "if the code and data so far are less than 510 bytes, pad
the remainder with zeros." This ensures the boot sector is exactly 512 bytes.

```
Offset    Contents
==========================================
0x000     start: (entry point)
  ...     Code + data
  ...     (zero padding)
0x1FE     0x55  <-- Boot signature
0x1FF     0xAA
==========================================
Total: 512 bytes (1 sector)
```

## 9. How to Build

```bash
cd i386/boot
make
```

Makefile contents:

```makefile
boot: boot.s
    as --32 -o boot.o boot.s        # Assemble in 32-bit mode
    ld -m elf_i386 --oformat binary -Ttext 0x0 -o boot boot.o
    rm -f boot.o
```

| Option | Meaning |
|:---|:---|
| `as --32` | Output 32-bit ELF (`.code16` generates 16-bit code) |
| `--oformat binary` | Output raw binary without ELF headers |
| `-Ttext 0x0` | Set the text section start address to 0 |

The generated `boot` (512 bytes) is ultimately concatenated at the beginning
of the floppy image:

```makefile
# Final line of i386/Makefile
cat boot/boot table start kernel > i386
```

## 10. Floppy Disk Image Structure

### 10.1 Image Composition

In the final step of the Makefile, four binary files are simply concatenated
with `cat` to create a single floppy image:

```makefile
# i386/Makefile
cat boot/boot table start kernel > i386
```

```
File         Contents            Size           Offset in Image
=====================================================================
boot/boot    Boot sector         512 bytes      0x0000 (sector 0)
table        GDT table           4096 bytes     0x0200 (sectors 1-8)
start        Second-stage boot   1024 bytes     0x1200 (sectors 9-10)
kernel       Kernel body         ~107 KB        0x1600 (sectors 11+)
=====================================================================
```

### 10.2 Why the Sizes Are Fixed -- The `.org` Directive

When concatenating with `cat`, if any file's size is off by even a single byte,
the placement of all subsequent files shifts. For this reason, boot, table, and
start all **explicitly fix their sizes using the `.org` directive**:

| File | `.org` Value | Size | Source |
|----------|------------|--------|--------|
| boot/boot | `.org 510` + `.word 0xAA55` | 512 bytes | `boot/boot.s:454-455` |
| table | `.org 4096` | 4096 bytes | `table.s` generated by `genasm.c:72` |
| start | `.org 1024` | 1024 bytes | `start.s:253` |
| kernel | (none -- natural size) | ~107 KB | Linker output converted by `elf` tool |

`.org N` fixes the output binary size to N bytes (if the actual code is less
than N bytes, the remainder is filled with zeros). This ensures that regardless
of how small each file's actual content is, the offsets after `cat` concatenation
are always consistent.

### 10.3 Memory Placement by the Bootloader

The BIOS loads sector 0 of the floppy (= boot/boot) to `0x7C00` and executes it.
boot.s reads sectors 1 through 299 sequentially starting at segment `0x0200`
(linear address `0x2000`). As a result, each concatenated file is placed at the
following linear addresses:

```
Offset in          Load Destination      Calculation
Image              Linear Address
--------------------------------------------------------------
0x0000 (boot)      0x7C00              Loaded by BIOS (sector 0)
0x0200 (table)     0x2000              0x2000 + 0x0000 = 0x2000
0x1200 (start)     0x3000              0x2000 + 0x1000 = 0x3000
0x1600 (kernel)    0x3400              0x2000 + 0x1400 = 0x3400
--------------------------------------------------------------
```

Here is how the calculation works:
- table (4096 bytes = 0x1000) is placed at 0x2000
- start comes right after table, so 0x2000 + 0x1000 = **0x3000**
- kernel comes right after start, so 0x3000 + 0x0400 (1024) = **0x3400**

### 10.4 Correspondence with the Linker's `-Ttext=0x3400`

The kernel's linker options are as follows:

```makefile
LDFLAGS = -m elf_i386 -n -T kernel.ld -static
```

The `. = 0x3400` in `kernel.ld` specifies that "the kernel code will be placed
at linear address 0x3400," and the linker resolves addresses based on this
assumption. Since the bootloader actually loads the kernel to 0x3400, the
linker's assumption matches the runtime placement.

**In other words, this entire mechanism is built on the following chain:**

1. `.org` fixes the sizes of boot (512B), table (4096B), and start (1024B)
2. `cat` concatenates them, determining the offsets within the image
3. boot.s reads sectors 1 onward to 0x2000, determining the linear address of each file
4. table -> 0x2000 (= `AL_GDT`), start -> 0x3000 (= `AL_KERNEL16`),
   kernel -> 0x3400 (= `-Ttext=0x3400`) -- everything is consistent

### 10.5 Running on QEMU

When running on QEMU, this image is written to a 1.44MB floppy
(`run.sh` creates it with `dd`):

```bash
dd if=/dev/zero of=floppy.img bs=512 count=2880    # Create an empty 1.44MB image
dd if=i386 of=floppy.img conv=notrunc              # Write the kernel image at the beginning
```
